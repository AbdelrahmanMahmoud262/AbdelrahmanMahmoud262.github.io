## Summary

Refactoring a live production application is one of the most high-stakes engineering challenges. The wrong approach — a hard rewrite — freezes feature delivery, accumulates divergence risk, and often results in a product that never ships. Taxi Alwatani, Iraq's primary ride-hailing platform serving 30,000+ active users, needed a complete architectural overhaul from a legacy structure to Clean Architecture, Jetpack Compose, and modern dependency injection — while maintaining uninterrupted production releases throughout the entire process.

This article documents the exact incremental strategy we used, the architectural boundaries we established, and the lessons learned leading a team of three Android developers through a full production refactor.

---

## Context: The Legacy Architecture

The legacy Taxi Alwatani passenger app had several structural problems:

- **Monolithic ViewModels** — Each ViewModel contained networking, business logic, and UI state management as a flat, unseparated blob
- **Static managers** — Singletons accessed directly from Activities and Fragments with no abstraction layer
- **XML layouts** — Mixed Compose/XML attempts with no coherent migration path
- **No DI framework** — Manual dependency construction with no lifecycle awareness
- **No unit tests** — Business logic was untestable because it was coupled directly to Android framework classes

The effect was a slow, fragile release cycle. Any change in one area had unpredictable ripple effects across the codebase.

---

## The Strategy: Layer-by-Layer Migration

We adopted a strict rule: **do not rewrite horizontally** (i.e., do not migrate one entire screen at a time). Instead, migrate **vertically by layer** — data layer first, domain layer second, presentation layer last.

This approach allowed every layer change to be independently shippable, testable, and rollbackable without affecting the other layers.

### Phase 1: Establish Domain Contracts

Before touching any networking or UI code, we defined clean domain contracts as Kotlin interfaces. These interfaces became the single source of truth for what each repository can do — with zero knowledge of how data is fetched or stored:

```kotlin
// domain/repository/RideRepository.kt
interface RideRepository {
    fun getActiveRides(): Flow<Result<List<Ride>>>
    fun getRideHistory(): Flow<Result<List<RideRecord>>>
    suspend fun requestRide(pickup: Location, dropoff: Location): Result<Unit>
    suspend fun cancelRide(rideId: String): Result<Unit>
}

// domain/repository/DriverRepository.kt
interface DriverRepository {
    fun getNearbyDrivers(location: Location, radiusKm: Double): Flow<Result<List<Driver>>>
    fun observeAssignedDriver(rideId: String): Flow<Driver?>
}
```

These domain interfaces exist in a pure Kotlin module (`:domain`) with zero Android dependencies. They can be unit tested with plain JUnit without a device or emulator.

### Phase 2: Implement Data Layer with Ktor + Hilt

With contracts defined, we replaced the legacy networking layer with **Ktor** for HTTP and WebSocket handling. Ktor was chosen over Retrofit because:

1. It handles both HTTP and WebSocket with a unified client
2. It's Kotlin-first with native coroutine support
3. It's significantly lighter than Retrofit + OkHttp for our use case

```kotlin
// data/remote/RideRemoteDataSource.kt
class RideRemoteDataSourceImpl @Inject constructor(
    private val httpClient: HttpClient
) : RideRemoteDataSource {

    override suspend fun requestRide(
        pickup: Location,
        dropoff: Location
    ): RideResponse {
        return httpClient.post("rides/request") {
            contentType(ContentType.Application.Json)
            setBody(RideRequestDto(
                pickupLat = pickup.latitude,
                pickupLng = pickup.longitude,
                dropoffLat = dropoff.latitude,
                dropoffLng = dropoff.longitude,
            ))
        }.body()
    }
}
```

Repository implementations map DTOs to domain models, keeping the domain layer free of network concerns:

```kotlin
class RideRepositoryImpl @Inject constructor(
    private val remoteDataSource: RideRemoteDataSource,
    private val localDataSource: RideLocalDataSource, // Room
) : RideRepository {

    override fun getActiveRides(): Flow<Result<List<Ride>>> =
        localDataSource.observeActiveRides()
            .map { entities -> Result.success(entities.map(RideEntity::toDomain)) }
            .catch { emit(Result.failure(it)) }
}
```

### Phase 3: Migrate to Jetpack Compose

With the data and domain layers isolated and tested, we began the UI migration. The key principle was **hybrid interoperability** — Compose and XML can coexist in the same Activity. This allowed us to migrate one screen at a time without freezing delivery of new features.

```kotlin
// Old: XML-based fragment
class RideStatusFragment : Fragment(R.layout.fragment_ride_status)

// New: Compose screen collected from a clean ViewModel
@Composable
fun RideStatusScreen(viewModel: RideViewModel = hiltViewModel()) {
    val uiState by viewModel.uiState.collectAsStateWithLifecycle()

    when (uiState) {
        is RideUiState.Loading -> LoadingScreen()
        is RideUiState.Active -> ActiveRideMap(uiState.ride)
        is RideUiState.Completed -> RideCompletedCard(uiState.record)
        is RideUiState.Error -> ErrorScreen(uiState.message)
    }
}
```

State is modeled as sealed classes, making every screen state explicit and exhaustive:

```kotlin
sealed class RideUiState {
    data object Loading : RideUiState()
    data class Active(val ride: Ride, val driver: Driver?) : RideUiState()
    data class Completed(val record: RideRecord) : RideUiState()
    data class Error(val message: String) : RideUiState()
}
```

This eliminates the classic `isLoading`, `errorMessage`, `data` triple-nullable state problem that plagues legacy ViewModels.

## Dependency Injection: Dagger Hilt Migration

Migrating from manual dependency construction to Dagger Hilt was done module by module. The trick is to keep both the old and new DI approaches alive during migration and switch screens individually.

```kotlin
@HiltAndroidApp
class TaxiAlwataniApp : Application()

// Each module declares its bindings
@Module
@InstallIn(SingletonComponent::class)
abstract class RepositoryModule {

    @Binds
    abstract fun bindRideRepository(impl: RideRepositoryImpl): RideRepository

    @Binds
    abstract fun bindDriverRepository(impl: DriverRepositoryImpl): DriverRepository
}
```

The strict separation between `@Singleton`, `@ActivityScoped`, and `@ViewModelScoped` components resolved several memory leaks that were previously silently occurring — ViewModels were holding Context references that outlived their Activity.

## Unit Testing: Domain Layer First

One of the primary motivations for Clean Architecture is testability. Once the domain layer was isolated in a pure Kotlin module, writing unit tests became straightforward:

```kotlin
// domain/usecase/RequestRideUseCaseTest.kt
class RequestRideUseCaseTest {

    private val rideRepository: RideRepository = mockk()
    private val useCase = RequestRideUseCase(rideRepository)

    @Test
    fun `requestRide returns success when repository succeeds`() = runTest {
        val pickup = Location(lat = 33.3, lng = 44.4)
        val dropoff = Location(lat = 33.5, lng = 44.6)

        coEvery { rideRepository.requestRide(pickup, dropoff) } returns Result.success(Unit)

        val result = useCase(pickup, dropoff)
        assertThat(result.isSuccess).isTrue()
    }
}
```

We established unit test coverage for all use cases in the domain layer within the first month. This became the safety net that made the rest of the refactor possible — any regression in business logic was caught before it could reach production.

## Leading the Team Through the Refactor

As Team Lead, the architectural refactor also required a team process refactor. Key decisions:

**Code review gates**: Every PR touching the domain or data layer required architectural review, not just logic review. We used PR templates with checklist items: "Does this change expose Android dependencies to the domain layer? Is this testable?"

**Feature-flag new screens**: New Compose screens were hidden behind a `BuildConfig` feature flag until they passed QA, allowing CI/CD to continue shipping the XML version to production.

**Architecture Decision Records (ADRs)**: Every significant decision (Ktor vs. Retrofit, Hilt vs. Koin) was documented in a lightweight ADR markdown file in the repository. This gave new team members context and prevented revisiting decided debates.

## FAQ

**Q: How long did the full refactor take?**
The data and domain layers were migrated in approximately 8 weeks. The full Compose UI migration is ongoing, with screens migrated incrementally alongside new feature delivery.

**Q: What's the risk of running Compose and XML in the same app?**
Minimal if you follow the interoperability guidelines. The main risk is inconsistent state management if you mix Fragment ViewModels with Compose ViewModels. We enforced that all state lives in `hiltViewModel()` regardless of whether the host is XML or Compose.

**Q: Why Ktor over Retrofit?**
Taxi Alwatani uses WebSockets extensively for real-time dispatch. Ktor handles both HTTP and WebSocket through a unified `HttpClient`, which simplified our networking layer significantly. Retrofit would have required a separate WebSocket library (OkHttp WebSocket), creating two network abstraction layers.

**Q: How did you avoid disrupting the product team during the refactor?**
By maintaining two parallel implementations behind feature flags and strictly never deleting legacy code until the new implementation was fully tested and QA-approved. The product team continued receiving weekly feature releases throughout the entire refactor.
